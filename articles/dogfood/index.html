dogfood
an agent-browser skill that uses your app like a real user. no scripts. no manual qa.

chris tate built a thing called dogfood. the name comes from the expression "eating your own dog food," which means using the product you built. but tate took it literally. his tool is an agent that navigates your web application the way a real user would, finds the things that break, and tells you about them. it is qa automation without the automation scripts. you point it at a url and it explores.

the traditional approach to web app testing is to write scripts that simulate user flows. you script the login, the checkout, the form submission, the navigation. then you run those scripts every time you deploy. the problem is that scripts only test what you thought to script. they do not find the edge case where a user clicks the back button after adding an item to cart, or refreshes the page during a payment, or opens the same form in two tabs. users are creative in ways that qa engineers, however experienced, are not.

dogfood solves this by not scripting anything. it uses a browser automation framework to load your application, then makes decisions about what to click, where to type, and which pages to visit based on the actual state of the dom. it treats your app like an adventure game. every page is a level. every link is a potential path. every form is a puzzle. the agent's goal is to explore as much surface area as possible and document anything that looks wrong.

explore
the agent loads your app and begins clicking every interactive element it can find. links, buttons, form inputs, dropdowns. it navigates through the full site graph, backtracking when it hits dead ends, prioritizing untested paths.
test edge cases
it intentionally does things users do but developers forget: rapid clicks, form resubmissions, navigation during async operations, invalid inputs, extremely long text entries. it finds race conditions that unit tests miss.
console checks
every page load is monitored for javascript errors, failed network requests, and uncaught exceptions. the agent captures console output and flags errors that do not prevent the page from rendering but indicate broken functionality.
capture
when the agent finds something wrong, it takes a screenshot and records a video of the interaction that led to the failure. the report includes the exact sequence of clicks, the dom state, and the console logs at the moment of failure.
severity
failures are categorized by impact. a javascript error on a static page is noted. a checkout flow that breaks is flagged critical. the report is structured so developers can triage without watching videos.

the architecture is elegant in its simplicity. the agent uses playwright or puppeteer as the browser layer, which means it runs a real chromium instance. it can test javascript-heavy single-page applications, server-rendered sites, and everything in between. the decision layer is a language model that reads the dom, decides what to interact with next, and evaluates whether the resulting state is correct. this is slower than scripted tests but covers more surface area.

the tradeoff is speed. a full exploratory run of a medium-sized application might take thirty minutes. a scripted test suite for the same app might take two minutes. but the scripted suite tests twenty flows that the developer imagined. the agent tests two hundred flows that no one imagined. for catching regressions in unexpected places, the agent wins. for validating that your checkout still works after a deploy, the script wins. the ideal setup is both: scripts for critical paths, agent for exploratory coverage.

dogfood is available as a skill that can be added to hermes agent or run standalone. the standalone version is a cli tool that takes a url and produces an html report. the skill version integrates into hermes' tool-calling loop, so you can ask your agent to "test my app" and it will spin up dogfood, run the exploration, and summarize the findings in the chat.

the long-term vision is a world where no one writes brittle end-to-end tests by hand. you describe your application in natural language, the agent explores it, and the resulting coverage map becomes your living test suite. when you deploy, the agent re-explores and compares against the baseline. anything that changed in an unexpected way is flagged. the tests update themselves because the tests are not scripts. they are behaviors.
