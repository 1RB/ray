docker on vercel

vercel now runs any dockerfile. if it speaks http, it deploys.
4 min read from @raysterminal

vercel built its name on zero-config frontend deploys. you push a next.js app, they read the framework, they derive the infrastructure, you get a url. no dockerfile, no build script, no yaml. that model works for most web apps, and it is still the fastest way to ship a frontend.

but there was always a gap. if you had a go service, a rails app, a spring boot api, or a web server behind nginx, you were on your own. you'd run it on a vps, or fargate, or fly.io, or render, or a kubernetes cluster you babysat on weekends. the frontend lived on vercel and the backend lived somewhere else, and the two talked over the public internet like distant cousins at a holiday dinner.

that gap is now closed. you add a file called Dockerfile.vercel to your project, run vercel deploy, and vercel builds the image, stores it in a project-level registry, ships it to their fluid compute platform, and hands you a production url. one command, one platform, one bill.

Dockerfile.vercel
# build stage
FROM golang:1.24-alpine AS build
WORKDIR /src
COPY . .
RUN go build -o /server main.go

# runtime stage
FROM alpine:3.20
COPY --from=build /server /server
CMD ["/server"]
terminal
$ vercel deploy
▲ Vercel CLI
✓ Building image from Dockerfile.vercel
✓ Stored image in your project's registry
✓ Deployed to Fluid compute
Production: https://my-server.vercel.app

the only contract is that your server listens on $PORT, which defaults to 80. if it speaks http, it deploys. rails, spring boot, express, laravel, asp.net, fastapi, nginx proxies, and anything else that opens a socket and returns a response. the blog post says "yes, even java" and "yes, even php," which reads like someone at vercel has been waiting a long time to type those words.

what you get is the same deployment model that frontends have had for years. every git push builds the image and produces an immutable preview url you can share, open, and roll back to. autoscaling works in both directions: traffic arrives and instances spin up, traffic stops and they wind down. you never size a fleet or pick a concurrency number.

the pricing model is what makes this interesting. vercel calls it active cpu pricing. fluid compute bills for the time your code is actually running, not the wall time the container is alive. a server parked on a slow database query or waiting on an upstream api response is not burning cpu while it waits. you pay for execution, not occupancy. for i/o-bound services this is materially cheaper than paying for a container that sits idle 90 percent of the time.

a container is now a first-class citizen, running on the same system as everything else.

the integration story is the real pitch. your container sits beside your frontend in the same project, on the same domain, and talks to it privately over the vercel network. logs, traces, and metrics for the container live in the same dashboard as your frontend. your full stack ships as a single deploy. this is the thing that fly.io and render got partially right, but neither had the frontend story. vercel had the frontend story and was missing the backend. now they have both.

startup speed is the question everyone asks about containers on serverless platforms. vercel addresses this by storing each image as an "optimized boot image," a compressed snapshot of the container's disk tuned for fast startup. when a container boots, they stream the snapshot and decompress on demand rather than downloading the full image before anything runs. the server can start handling requests before the entire image is in place. once an instance is running, fluid compute keeps it warm and serves many requests from it, so you get the responsiveness of a long-lived server with the billing profile of one that sleeps when idle.

containers are stateless processes. they take a request, return a response, and keep nothing in between. persistent state lives in a backing service you attach, like a database or cache from the vercel marketplace. because an instance holds nothing that has to survive, vercel can add instances when traffic arrives and retire them when it stops. the blog mentions durable storage attached to containers is coming, which would unlock stateful workloads, but for now the model is stateless-first.

the "why now" framing in the blog is honest. vercel's first platform let you deploy a dockerfile with a single command a decade ago, but the infrastructure to make it great didn't exist yet. they spent the years since building the primitives that power builds, functions, sandboxes, and now containers. framework detection remains the front door for most apps, because when vercel recognizes your framework, they can read your code and derive the infrastructure automatically. a dockerfile is the escape hatch for everything else: a service that needs a system library like ffmpeg or chromium, a framework they don't auto-detect yet, or an app you want to bring exactly as it already runs.

build
vercel reads Dockerfile.vercel and builds the image automatically. no daemon, no local docker, no registry setup.
deploy
image ships to fluid compute. one command. production url. every push gets a preview.
scale
autoscaling in both directions. no fleet sizing, no concurrency tuning. instances arrive with traffic and retire when it stops.
price
active cpu pricing. you pay for execution time, not wall time. idle containers waiting on i/o don't burn money.
observe
logs, traces, and metrics in the same dashboard as your frontend. one platform, one view.
network
container talks to your frontend privately over the vercel network. same project, same domain, one deploy.

the positioning is clear. this is not "vercel is now a generic container host." it is "your backend now ships the way your frontend does." framework detection handles the 90 percent case where vercel can infer everything from your code. the dockerfile handles the 10 percent where they can't, or where you don't want them to. the two paths converge on the same platform, the same billing, the same observability, and the same deployment workflow.

for teams already on vercel, this removes the last reason to run a separate backend somewhere else. for teams not on vercel, the pitch is simpler than it used to be: bring a dockerfile, get the same deploy experience for your server that next.js users have had for years. whether that's enough to pull people off fly.io, render, or their own kubernetes cluster depends on workload, pricing, and how much they care about having frontend and backend in one place. but the gap that made vercel a frontend-only platform is gone.
