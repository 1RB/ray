cobe

a five kilobyte webgl globe library for landing pages. ultra-light. customizable. smooth as hell.
4 min read from @raysterminal

shuding built a globe that weighs five kilobytes. not fifty. not five hundred. five. after gzip. this is a complete, interactive, hardware-accelerated three-dimensional earth that you can spin, zoom, and customize, in a package smaller than most hero images. the average web developer's ego is larger than this library.

npm i cobe

the implementation is a single webgl context rendering a sphere with a custom shader. the shader handles the atmosphere glow, the night lights, the rotation, and the mouse interaction. there is no external dependency on three.js, babylon, or any other 3d framework. the entire library is self-contained because shuding wrote the glsl by hand and optimized the geometry to the minimum viable sphere. this is the opposite of the modern javascript ecosystem, where a simple modal dialog pulls in seventeen transitive dependencies and a twelve-megabyte bundle.

the api is minimal by necessity. you create a canvas, pass it to cobe, and configure options. marker positions, colors, rotation speed, atmosphere thickness. the defaults are tasteful: a dark earth with city lights, subtle atmosphere, slow rotation. it looks expensive. it is free. you can drop it onto any landing page and immediately signal global presence, international reach, or just good taste. the rotation is smooth because it runs on the gpu. the interaction is smooth because the event handling is throttled correctly. the bundle is small because there is no framework abstraction layer between the code and the gpu.

the use cases are specific. this is not a general-purpose mapping library. it is a decoration. it exists to make a landing page feel more considered. a saas company with customers in multiple countries. a conference with international speakers. a developer tool with a global user base. the globe says "we are everywhere" without requiring a mapbox api key or a google maps integration or a ten-second load time.

what makes cobe notable is not just the size. it is the proof that webgl does not require a framework. developers have been taught that 3d on the web means importing three.js, setting up a scene, a camera, a renderer, lights, materials, and then fighting the framework to get what you actually want. shuding skipped all of that. he wrote the shaders he needed, the geometry he needed, and nothing else. the result is faster, smaller, and more reliable than any framework-based equivalent because there is no framework to break.

the customization options are exposed through a simple javascript object. you can change the base color of the earth, the color of the markers, the intensity of the atmosphere, the speed of rotation, and the zoom level. you can add markers by passing latitude and longitude coordinates. the markers glow. the earth rotates. the atmosphere diffuses light at the edges. it is the same visual language that apple and stripe use on their landing pages, except apple's globe is a video file and stripe's is a custom webgl implementation that probably took an engineer six months. cobe took shuding a weekend, and it is available to anyone who types npm install.

the library is open source and the license is permissive. you can use it commercially, modify it, or learn from it. the repository includes a live demo and documentation. the code is readable. the shaders are commented. this is rare in a javascript ecosystem that increasingly treats code as opaque infrastructure. you can read cobe's source and understand exactly how it works. this is part of the point. the minimal size is not just a performance optimization. it is a statement about what is necessary versus what is assumed.

if you have a landing page that needs to feel global, and you do not want to add a framework, and you do not want to load a video, and you do not want to pay for a mapping api, cobe is the answer. five kilobytes. one canvas. no dependencies. smooth as hell. this is what web development looks like when someone decides to care about the user before they care about the developer experience.
