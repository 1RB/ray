zed 1.0

the ide hits version one. what they built, what they skipped, and whether it is enough.
5 min read from @raysterminal

zed announced version 1.0 after years of public beta. the ide is built in rust with a custom gpu-accelerated rendering engine. it is fast. opening a million-line repository is instant. the ui is minimal, the latency is low, and the memory footprint is smaller than electron-based competitors. the tradeoff is that extensions are limited, the ecosystem is small, and many features users expect from mature ides are still missing.

the 1.0 release focused on stability and collaboration. zed has real-time multiplayer editing built in, not bolted on. multiple users can edit the same file with cursors visible, without the lag that makes google docs feel heavy in a codebase. this is the feature zed's founders, who previously built atom at github, care about most. they believe coding is collaborative and the tools should reflect that.

the ai integration is native, not a sidebar plugin. you can generate, edit, and explain code inline. the model runs in the cloud but the interface is integrated into the editor's command palette. there is no chat window taking up screen space. the model's output streams directly into the buffer. this sounds small but it changes the rhythm of ai-assisted coding from "switch contexts, paste, clean up" to "ask, receive, continue."

what is missing: a debugger, a robust extension marketplace, support for remote development over ssh, and native support for many languages beyond the core set. zed's position is that they will add these, but 1.0 means the core is solid, not that the product is finished. the risk is that users who need a debugger today will not wait. the opportunity is that users who are tired of slow, heavy ides will switch for speed and stay for the features as they arrive.

zed 1.0 ships without a debugger and with a limited extension ecosystem. if you need those today, this is not your ide yet. the bet is that speed and collaboration will convert users before the feature gap closes.

the business model is subscription-based for teams, free for individuals. this is the inverse of vscode, which is free for everyone and monetizes through microsoft's cloud services. zed needs to convince companies that real-time collaboration is worth paying for, and that the speed difference is meaningful enough to overcome the friction of switching. it is a hard pitch. developers are loyal to their tools and the switching cost is high.

whether zed succeeds depends on whether the developer tools market has room for a new commercial ide. vscode dominates. jetbrains is entrenched. vim and emacs never die. zed's bet is that performance and collaboration are differentiators large enough to carve out a space. 1.0 is the starting line, not the finish. the next year will determine if anyone shows up to race.
