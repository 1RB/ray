zed 1.0

the code editor built by the atom team, now out of beta

the team that built atom, github's first editor, has spent the last four years building its replacement. zed is now at version 1.0, and it is a different kind of code editor. multiplayer by default. gpu-rendered. written in rust. and fast in a way that electron-based editors cannot touch.

the multiplayer implementation is the headline feature. zed opens every project as a shared workspace. you can invite collaborators by link, see their cursors in real time, and edit the same file simultaneously without the lag that kills flow in other tools. the team calls it "channel-based" collaboration: a persistent shared context that includes not just the code but the terminal, the ai chat, and the file tree. when you join a channel, you join the entire environment, not just a single buffer.

"zed treats coding as a multiplayer game by default. every project is a shared world."

the rendering engine is built on gpui, a custom rust framework that draws the entire interface with the gpu. this means no dom, no css, no javascript in the rendering path. the result is consistent frame times, no jank when scrolling large files, and crisp text at any scale. on a modern mac, zed scrolls a million-line file without dropping a frame. this is not a benchmark trick. it is the baseline experience.

rust is not just an implementation detail. it shapes what the editor can do. memory safety without garbage collection means zed can spawn threads freely, parallelize syntax highlighting across cores, and keep the ui responsive while background tasks run. the language also enforces discipline: the zed codebase has fewer classes of bugs by construction, which matters when you are building infrastructure that developers live inside for hours every day.

the ai integration is native, not bolted on. zed has a built-in assistant panel that can reference the current file, the current project, or the entire codebase. it uses the context to generate, explain, and refactor code. because the assistant runs in the same process as the editor, it has direct access to the abstract syntax tree, not just the text. this means it can perform structural operations like "extract this into a function" or "rename this variable across the project" with semantic accuracy.

the extension model is still evolving. zed 1.0 ships with support for language servers and tree-sitter grammars, but the full plugin api is not yet open. the team has committed to releasing it, and the architecture suggests it will be wasm-based, meaning extensions run in a sandboxed environment without direct access to the editor internals. this is slower to ship than a javascript api but safer for users.

the pricing model is unusual for a developer tool. zed is free for individuals and open source projects. teams pay per seat for the multiplayer features, with the exact pricing tiered by organization size. the ai assistant has its own usage-based pricing, though you can bring your own api key and pay the provider directly. the team is explicit that they do not want to be an ai company. the editor is the product. the ai is a feature.

there are limitations. zed is mac-only at 1.0, with linux and windows builds in progress. the extension ecosystem is smaller than vscode's by orders of magnitude. and the learning curve is real if you are coming from an electron editor: the keybindings are different, the settings are configured in json rather than a gui, and the visual language is minimal to the point of austerity.

but if you value speed, if you work in teams, or if you are tired of editors that feel like web apps, zed is the most credible alternative to have launched in years. the atom team learned from their first product. they know how electron editors fail, and they built the opposite.
